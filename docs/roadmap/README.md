# Roadmap — 投研 + 量化投资系统

> 创建：2026-06-30 ｜ 视角：从「投研 + 量化」专业角度重排的落地路线图
> 本目录是 `docs/audit-and-redesign-2026-06.md`（架构审计）之后的**方向修正版**：审计回答了「现状缺什么」，本 roadmap 回答「作为一个投研系统，应该往哪走、按什么顺序走、为什么」。

---

## 0. 一句话定位

> **这套系统的护城河不是「又一个量化回测引擎」——以你几十只关注股的样本量，在那个赛道上统计意义上赢不了；护城河是「把我的真实交易决策、当时的市场状态、事后结果，闭环起来做归因」。这是只有你自己的数据才能做、且绝大多数工具都没有的东西。**

工程重心应压在这个**闭环**上，而不是平铺去追求「全自动 alpha 工厂」。系统定位为**决策支持 + 纪律审计**，不做自动下单（这一点现状已经做对）。

## 2. 与旧 roadmap 的关键差异（自我修正）

第一版 roadmap（对话中给出）有几个量化新手易犯的错，本目录已纠正：

| 旧版主张 | 问题 | 修正 |
|----------|------|------|
| 「回测引擎是核心分水岭」 | 几十只股 + 判别式打分做组合回测，样本量不足 + 幸存者偏差 + 过拟合，结论是噪声 | **回测降级**：事件研究（A）优先 → 横截面 IC（B）次之 → 全组合回测（C）最后且谨慎。见 [00](00-principles-and-direction.md#二把回测拆成三个严肃度层级) |
| 「用回测结果反推权重」 | 这是在**保证过拟合**（数据窥探 / 多重检验） | 删除该做法；权重只在样本外、跨区制稳健时才微调 |
| 「FRED 宏观排在很靠前」 | 宏观择时历史战绩差，CPI/就业是滞后数据，对择时基本是噪声 | **市场广度 > HY 信用利差 > 收益率曲线 >> CPI/就业**；优先用已有价格数据算广度 |
| 「计划」与「复盘」是两条独立主线 | 它们是同一个闭环的两端 | 合并为 **Phase 2 计划-执行-复盘闭环**，共享主键 |
| 复盘 = 给单笔交易写点评 | 价值低 | 复盘 = **跨笔行为模式识别** + 实盘期望值 + 三方对比 |
| （缺失） | 没有组合/敞口视角 | 新增：组合敞口是投研第一视角，见 [05](05-cross-cutting.md) |
| （缺失） | LLM 输出无法问责 | 新增：**LLM 校准追踪**（Brier score），见 [04](04-phase3-review-and-regime.md) |

## 3. 阶段总览与依赖

```
Phase 0  数据质量地基 + 事件研究          ← 最先做，最小成本验证「信号有没有用」
   │      (复权正确性 / point-in-time 取价 / 事件收益分布)
   ▼
Phase 1  数据地基（精简）                  ← 只补被事件研究证明有用的数据
   │      (OHLCV 落库 / 市场广度 / EPS surprise 结构化)
   ▼
Phase 2  计划-执行-复盘闭环 ★核心          ← 你三个目标的真正连接点
   │      (execution_plans + trade_journal 共享主键 / 计划遵守度 / 实盘期望值)
   ▼
Phase 3  复盘引擎 + 区制条件化            ← 复盘火力 + 让结论可执行
          (跨笔行为模式 / 三方对比 / regime 分层 / LLM 校准)

横切贯穿: 组合敞口视角 · 可视化 · 文档对齐 · 安全 · CI   （见 05）
随时可插: quick-wins backlog                              （见 06）
```

**为什么是这个顺序**：Phase 0 用最低成本回答「我关注的信号到底有没有边际信息」——如果没有，后面的交易计划和复盘就只是把噪声包装成叙事。先证伪，再投入。Phase 2 之所以排在全组合回测之前，是因为「计划-执行-复盘闭环」用的是**你自己的真实交易数据**（无幸存者偏差、样本就是你关心的全部），价值密度远高于在小样本上做组合回测。

## 5. 落地纪律（沿用仓库现有约定）

- **TDD**：新增逻辑必须先写测试；外部依赖（DB / 网络 / LLM）全部 mock 或注入，离线可跑。
- **分层**：`api/routes/*` → `services/*` → `db.py` / `tasks/*`，路由层薄封装。
- **优雅降级**：无 `INVESTMENT_ASSISTANT_DATABASE_URL` 时只读端点返回空 + `{"degraded": true}`，不崩。
- **不静默吞错**：外部失败结构化返回 + 日志，不裸 `except` 后丢弃。
- **配置已预留**：`config.py` 已定义 `MacroConfig` / `FundamentalsConfig` / `PriceConfig` / `StrategyParams` / `BacktestConfig` 脚手架（当前未实现），实施时填充即可，无需重设计配置结构。

## 6. 待你确认的输入（会影响数据模型细节）

这些信息能让 Phase 2/3 的数据模型更贴合你的实际，建议在进入 Phase 2 前补充到 [03](03-phase2-plan-execution-loop.md) 的「待定」区：

1. **交易记录现状**：你现在怎么记交易？（Obsidian 笔记 / 券商 CSV / 纯脑子）——决定 `trade_journal` 的录入方式（表单 / 导入 / 解析）。
2. **标的池规模与风格**：watchlist 当前默认 `CRDO/MU/RKLB/NVDA`（半导体+成长/趋势）。你实际池子多大、主要风格是波段 / 趋势 / 事件驱动？——决定事件研究的样本聚合方式与区制定义。
3. **复盘关注点**：你最想被系统「揪出来」的坏习惯是什么？（追高 / 过早止盈 / 死扛亏损 / 高波动时过度交易）——决定 Phase 3 行为模式检测的指标。
